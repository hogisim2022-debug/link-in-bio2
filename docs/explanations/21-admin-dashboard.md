## 무엇을 했나?

링크별 클릭 수를 시간 기준으로 집계하는 대시보드를 만들었다.

```
Admin 대시보드 영역
┌─────────────────────────────────────────┐
│  총 클릭 수            [7일] [30일]      │
│  42                                     │
│─────────────────────────────────────────│
│  링크           클릭   비율             │
│  카카오 채널      28    67%             │
│  이메일 문의      10    24%             │
│  강의 소개서       4     9%             │
└─────────────────────────────────────────┘
```

### 왜 새 테이블이 필요했나?

`links` 테이블의 `click_count`는 누적 숫자만 저장한다. "최근 7일 클릭 수"를 계산하려면 **언제** 클릭했는지 기록이 필요하다.

```
기존 구조 (누적 카운터)       새 구조 (이벤트 기록)
links 테이블                  link_click_events 테이블
┌─────────────────┐           ┌────────────────────────────┐
│ id              │           │ id                         │
│ title           │           │ link_id  → links.id        │
│ click_count: 42 │           │ user_id                    │
└─────────────────┘           │ clicked_at: 2026-04-27...  │
                               └────────────────────────────┘
                               (클릭할 때마다 행 1개 추가)
```

---

## 핵심 개념 1: 이벤트 테이블 패턴

숫자 하나로 저장하는 것을 "집계(aggregate)", 행 하나씩 기록하는 것을 "이벤트(event)"라고 한다.

| 방식 | 저장 형태 | 할 수 있는 것 |
|---|---|---|
| 집계 (`click_count`) | 숫자 1개 | 전체 합계만 |
| 이벤트 (`link_click_events`) | 클릭마다 행 1개 | 기간별, 시간대별 분석 |

이벤트 방식은 저장 공간을 더 쓰지만, 분석 유연성이 훨씬 높다.

- ❌ 과거: `"최근 7일 클릭 수 보여줘"`
- ✅ 현재: `"클릭할 때마다 이벤트 기록하고, 대시보드에서 기간 필터로 집계해줘"`

---

## 핵심 개념 2: RPC 업데이트

기존 `increment_click_count` 함수가 `click_count`를 1 올리는 것에 더해, 이벤트 테이블에도 행을 삽입하도록 수정했다.

```
링크 클릭
   ↓
increment_click_count(link_id) 호출
   ↓
   ├── links.click_count + 1     (누적 카운터 유지)
   └── link_click_events에 행 삽입  (시간 기록 추가)
```

두 작업이 하나의 RPC 안에 있어서, 하나가 실패하면 둘 다 취소된다.

---

## 핵심 개념 3: 클라이언트 컴포넌트에서 데이터 페칭

기간 필터 버튼을 누를 때마다 Supabase에 새 요청을 보내서 해당 기간의 이벤트를 불러온다.

```
[7일] 버튼 클릭
   ↓
period 상태 변경: "30d" → "7d"
   ↓
Supabase: clicked_at >= 7일 전인 이벤트 조회
   ↓
링크별로 개수 세기 (클라이언트에서)
   ↓
테이블 렌더링
```

- ❌ 과거: `"기간 필터 버튼 만들어줘"`
- ✅ 현재: `"7일/30일 버튼을 누르면 Supabase에 재요청해서 해당 기간 이벤트만 가져오게 해줘"`

## 퀴즈 정답

1. 기간 필터(7일/30일)를 구현하려면 기존 `click_count` 컬럼만으로 충분하다. → **X**  
   ↳ `click_count`는 누적 숫자만 저장한다. 기간별 집계에는 클릭 시각이 기록된 `link_click_events` 테이블이 필요하다.

2. `link_click_events` 테이블의 RLS SELECT 정책은 비로그인 방문자도 읽을 수 있게 설정한다. → **X**  
   ↳ SELECT는 `auth.uid() = user_id` — 로그인한 본인만 읽을 수 있다. 클릭 통계는 관리자 전용이다.

3. 기간 필터 버튼을 누르면 서버에 새 요청을 보내지 않고 이미 받아둔 데이터에서 필터링한다. → **X**  
   ↳ 버튼을 누를 때마다 Supabase에 새 요청을 보낸다. `period` 상태 변경을 감지해 재요청한다.

4. `increment_click_count` RPC를 업데이트해서 `link_click_events`에도 기록을 남긴다. → **O**  
   ↳ RPC 하나가 `click_count + 1`과 이벤트 삽입을 동시에 처리한다.
