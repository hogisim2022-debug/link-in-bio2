## 핵심 개념 1: Storage 파일 삭제

Storage에서 파일을 삭제하려면 전체 URL이 아니라 **버킷 안의 경로**만 필요하다.

DB 행 삭제와 Storage 파일 삭제를 둘 다 해야 완전히 제거된다. 하나만 하면 DB에는 없지만 파일은 남거나, 반대로 파일은 없지만 DB에 행이 남는 불일치가 생긴다.

- ❌ 과거: `"로고 삭제 버튼 만들어줘"`
- ✅ 현재: `"삭제 버튼 누르면 Storage 파일과 DB 행을 둘 다 제거해줘"`

---

## 핵심 개념 2: 드래그 순서 변경과 DB 업데이트 타이밍

드래그를 놓는 순간(`onDragEnd`) 두 가지 일이 동시에 일어난다.

```
드래그 종료
   ↓
1. 화면 즉시 업데이트 (로컬 상태)
2. Supabase에 order 값 일괄 업데이트 (백그라운드)
```

화면이 먼저 바뀌고 DB는 조금 늦게 따라온다. 사용자는 지연 없이 결과를 볼 수 있다.

- ❌ 과거: `"드래그 후 저장 버튼 눌러야 순서 저장되게 해줘"`
- ✅ 현재: `"드래그가 끝나면 화면은 즉시 바꾸고, DB 업데이트는 백그라운드에서 처리해줘`

## 퀴즈 정답

1. 로고를 추가할 때 기업 이름 입력과 이미지 선택을 별도 단계로 나눈 이유는 이름 없이 업로드하면 파일만 Storage에 올라가고 DB에는 기록이 안 되기 때문이다. → **O**  
   ↳ 이름이 비어 있으면 `handleAdd` 함수가 조기 종료된다. DB에 이름이 있어야 미리보기 슬라이더에 표시된다.

2. 로고를 삭제하면 Storage 파일과 DB 행이 동시에 제거된다. → **O**  
   ↳ `handleDelete`에서 `storage.remove()`와 `client_logos.delete()`를 함께 호출한다.

3. 드래그로 순서를 바꾸면 화면은 즉시 변하지만 DB 업데이트는 드래그가 끝난 후 일어난다. → **O**  
   ↳ `onChange(reordered)`로 화면을 먼저 업데이트하고, 그 후 Supabase에 order 값을 일괄 업데이트한다.
