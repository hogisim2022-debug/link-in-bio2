## 이 문서를 읽고 나면 풀 수 있어요

1. Middleware는 페이지 렌더링보다 먼저 실행된다.
2. 브라우저 쿠키는 서버가 지우기 전까지 만료 시간 없이 영구적으로 유지된다.
3. 세션은 사용자의 로그인 상태를 서버가 기억하는 정보다.

---

## 세션(session)
사용자가 로그인 상태인지 서버가 기억하고 있는 정보

1. 로그인하면 서버 처리 순서
- 로그인 성공
- 서버가 세션 ID를 생성
- 그걸 브라우저 쿠키에 저장
- 이후 요청마다 쿠키가 같이 보내짐
- 서버는 “아 이 사람 로그인 했던 사용자구나” 판단

2. 로그아웃하면 서버가 쿠키를 삭제  
- 서버는 “이 사용자 더 이상 로그인 안 됐어”라고 판단  
- 다시 /login으로 보냄

## 브라우저 쿠키(cookie)
웹사이트가 사용자의 브라우저에 저장하는 작은 데이터 조각
> “이 정보 좀 네 컴퓨터에 잠깐 맡겨둘게. 다음에 다시 올 때 가져와.”

```
🔐 로그인 유지 (세션 ID 저장)
🌐 사용자 설정 기억 (언어, 다크모드 등)
🛒 장바구니 정보
📊 방문 기록 / 트래킹
```

### 동작 방식
```
1. 서버가 응답할 때 쿠키를 내려줌
2. 브라우저가 쿠키 저장
3. 이후 같은 사이트 요청할 때마다 자동으로 쿠키 같이 보냄
```

### 세션이랑 관계
- 쿠키 안에는 보통 세션 ID가 들어 있음
- 서버는 그 ID로 로그인 상태를 판단함

### 특징
- 용량 작음 (보통 몇 KB)
- 만료 시간 설정 가능 (자동 로그아웃)

## 핵심 개념 1: Next.js Middleware

Middleware는 요청이 페이지에 도달하기 **전에** 실행되는 코드다.  
브라우저가 `/admin`을 열려고 하면, 페이지를 렌더링하기 전에 middleware가 먼저 실행된다.

```
브라우저 요청
  └── /admin/* 경로?
       ├── 세션 있음 → 페이지 열림
       └── 세션 없음 → /login으로 이동
```

적용 경로: `/admin`, `/admin/settings` 등 /admin 하위 전체

- ❌ 과거: `"로그인 안 한 사람 어드민 못 들어오게 해줘"`
- ✅ 현재: `"middleware.ts에서 /admin 경로 접근 시 세션 없으면 /login으로 redirect해줘"`

## 핵심 개념 2: 현재 상태 (개발용 bypass)

지금은 `.env.local`에 Supabase URL이 없어서 middleware가 세션 확인 없이 통과시킨다.  
`.env.local`에 Supabase URL을 추가하면 실제 세션 검증이 작동한다.

## 핵심 개념 3: 로그인 폼 상태 관리

로그인 폼은 4개의 상태를 가진다.
- `email`, `password`: 입력값
- `error`: 실패 메시지 (잘못된 비밀번호 등)
- `loading`: 제출 중 버튼 비활성화 + 텍스트 변경

`handleSubmit`은 현재 개발용 더미 로직이다.  
(`admin@example.com` / `password` 로 로그인하면 `/admin`으로 이동)  
Supabase 연동 후 `supabase.auth.signInWithPassword()`로 교체한다.

## 퀴즈 정답

1. Middleware는 페이지 렌더링보다 먼저 실행된다. → **O**  
   ↳ 요청이 페이지에 도달하기 전에 실행되어 인증 체크에 활용한다.

2. 브라우저 쿠키는 서버가 지우기 전까지 만료 시간 없이 영구적으로 유지된다. → **X**  
   ↳ 쿠키에는 만료 시간을 설정할 수 있다. 만료되면 자동으로 삭제된다.

3. 세션은 사용자의 로그인 상태를 서버가 기억하는 정보다. → **O**  
   ↳ 로그인 시 서버가 세션 ID를 생성하고 쿠키에 저장해 이후 요청을 식별한다.
