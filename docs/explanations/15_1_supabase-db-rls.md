# 14. Supabase DB 테이블 & RLS

## 이 문서를 읽고 나면 풀 수 있어요

1. RLS를 설정하면 로그인한 사용자만 자신의 데이터를 수정할 수 있도록 제한할 수 있다.
2. 방문자도 링크를 클릭하면 `click_count`를 직접 UPDATE할 수 있다.
3. 마이그레이션은 DB 구조 변경을 코드로 기록해 이력을 추적하는 방법이다.

---

## 무엇을 했나?

이 서비스가 실제로 데이터를 저장하고 꺼내 쓸 수 있도록, **Supabase 데이터베이스에 표(테이블) 3개를 만들고 보안 규칙을 설정**했다.

엑셀에 비유하면 이렇다. 지금까지는 `mock-data.ts`라는 파일에 데이터를 직접 하드코딩해 뒀는데, 이번 작업으로 그 데이터가 실제 데이터베이스 서버로 이사할 준비가 된 것이다.

## 핵심 개념 : 테이블과 관계

하나의 데이터베이스 안에 목적별로 테이블을 나눈다.  
이 프로젝트의 테이블 3개가 서로 어떻게 연결되는지:




```
[로그인한 사용자 계정]
        │
        ├─ 하나의 프로필을 가진다 ──────► [profile 테이블]
        │                                   이름, 한 줄 소개, 경력 텍스트
        │                                   프로필 이미지 URL
        │                                   SEO 제목, 설명, OG 이미지 URL
        │                                   대표 CTA 링크 (아래 links 참조) ──┐
        │                                                                   │
        ├─ 여러 개의 링크를 가진다 ─────► [links 테이블]  ◄────────────────────┘
        │                                   링크 제목, URL, 종류(기본/카드)
        │                                   노출 여부, 클릭 수, 정렬 순서
        │                                   이미지 URL, 등록일시
        │
        └─ 여러 개의 기업 로고를 가진다 ► [client_logos 테이블]
                                            로고 이름, 이미지 URL, 정렬 순서
```

```
[profile 테이블]           [links 테이블]
┌─────────────────┐       ┌──────────────────────┐
│ id  (계정 ID)   │──┐    │ id                   │
│ name            │  │    │ user_id ─────────────┼──► profile.id와 같은 값
│ title           │  └───►│ title                │
│ primary_cta_    │       │ url                  │
│   link_id ──────┼──────►│ (links.id 참조)      │
│ seo_title       │       └──────────────────────┘
└─────────────────┘
                           [client_logos 테이블]
                          ┌──────────────────────┐
                          │ id                   │
                          │ user_id ─────────────┼──► profile.id와 같은 값
                          │ name                 │
                          │ image_url            │
                          └──────────────────────┘
```

## 테이블 설계 설명

### profile table — "내 소개 카드"

관리자 한 명의 프로필 정보를 저장한다. 행이 딱 1개만 존재한다.

| 컬럼 | 설명 |
|------|------|
| `id` | 로그인 계정 ID와 동일. 이 표의 기본 키 |
| `name` | 이름 |
| `title` | 직함·한 줄 소개 |
| `bio` | 소개 본문 |
| `experience` | 경력 텍스트 (예: "15년 경력") |
| `image_url` | 프로필 사진 URL |
| `primary_cta_link_id` | 하단 떠 있는 CTA 버튼에 연결할 링크 ID |
| `seo_title` | 검색엔진에 표시될 페이지 제목 |
| `seo_description` | 검색엔진에 표시될 설명 문구 |
| `og_image_url` | 카카오톡/SNS 공유 시 미리보기 이미지 URL |


### links table — "링크 목록"

공개 페이지에 표시할 링크들을 저장한다. 링크를 추가할 때마다 행이 늘어난다.

| 컬럼 | 설명 |
|------|------|
| `id` | 링크 고유 번호 |
| `user_id` | 이 링크를 소유한 계정 ID |
| `title` | 링크 제목 |
| `url` | 이동할 URL |
| `type` | 링크 종류 (`basic` 일반형 / `overflow` 돌출 카드형) |
| `image_url` | 카드형일 때 표시할 이미지 |
| `order` | 화면에 표시되는 순서 (숫자가 작을수록 위) |
| `is_visible` | 공개 여부 (false면 방문자에게 숨김) |
| `click_count` | 누적 클릭 수 |
| `created_at` | 링크 등록 일시 |

### client_logos table — "협력사 로고 슬라이더"

공개 페이지 Trust Bar 영역에 표시할 기업 로고들을 저장한다.

| 컬럼 | 설명 |
|------|------|
| `id` | 로고 고유 번호 |
| `user_id` | 소유 계정 ID |
| `name` | 기업 이름 (alt 텍스트용) |
| `image_url` | 로고 이미지 URL |
| `order` | 슬라이더 표시 순서 |

## RLS (Row Level Security) — "행 단위 자물쇠"

RLS는 **누가 어떤 행을 읽거나 수정할 수 있는지**를 데이터베이스 수준에서 통제하는 보안 기능이다.

비유하면, 엑셀 파일을 인터넷에 공개했을 때, "누구나 볼 수는 있지만 수정은 파일 주인만 가능"하도록 잠금을 거는 것과 같다.

| 동작 | 누가 할 수 있나? | 이유 |
|------|-----------------|------|
| **읽기(SELECT)** | 모든 방문자 | 공개 페이지(`/`)에서 로그인 없이 프로필·링크를 봐야 하기 때문 |
| **추가(INSERT)** | 로그인한 본인만 | 다른 사람이 내 링크를 멋대로 추가하는 걸 막기 위해 |
| **수정(UPDATE)** | 로그인한 본인만 | 내 데이터는 내가만 바꿀 수 있어야 하기 때문 |
| **삭제(DELETE)** | 로그인한 본인만 | 위와 같음 |

Supabase가 로그인 시 발급하는 `auth.uid()` 값이 존재.  
"본인 확인"은  이 값이 행의 `user_id`(또는 `profile.id`)와 일치하면 허용, 다르면 차단한다.

- ❌ 과거: `"내 데이터만 수정할 수 있게 해줘"`
- ✅ 현재: `"links 테이블 UPDATE/DELETE RLS 정책을 auth.uid() = user_id인 경우만 허용으로 설정해줘"`

## RPC 함수 `increment_click_count` — "클릭 수 올리기"
RPC = Remote Procedure Call
> “DB 서버에 있는 함수를 마치 내 로컬 함수처럼 호출하는 방식”

공개 페이지에서 방문자가 링크를 클릭하면 해당 링크의 `click_count`를 1 올려야 한다.

문제가 하나 있다. 방문자는 로그인을 하지 않은 상태이므로 RLS에 의해 원칙적으로 links 표를 수정할 수 없다. 그래서 **`increment_click_count`라는 특수 함수**를 만들었다.

이 함수는 `SECURITY DEFINER` 속성을 가진다. 쉽게 말해 "이 함수를 실행할 때만은 RLS 자물쇠를 무시하고 카운트만 올린다"는 뜻이다. 수정 권한은 없지만 카운트 집계만 할 수 있는 전용 창구를 만든 셈이다.

방문자가 링크를 클릭 → 프론트엔드가 `increment_click_count(링크ID)` 호출 → DB에서 해당 링크의 `click_count + 1` 실행.

- ❌ 과거: `"클릭 수 올려줘"`
- ✅ 현재: `"방문자는 RLS 때문에 links 직접 UPDATE 못 하니까 increment_click_count(링크ID) RPC 호출해줘"`

## 마이그레이션이란?

데이터베이스 구조 변경(표 만들기, 수정, 삭제 등)을 코드로 기록해 두는 것이다. 나중에 "언제 어떤 표를 만들었는지" 이력을 추적할 수 있다.

이번 작업에서 두 개의 마이그레이션을 Supabase에 적용했다.

1. `create_core_tables` — 표 3개 생성 + RLS 보안 규칙 설정
2. `create_increment_click_count_rpc` — 클릭 수 집계 함수 생성

## 이렇게 확인하세요

1. [supabase.com](https://supabase.com) 접속 → 프로젝트 선택
2. 좌측 메뉴 **Table Editor** → `profile`, `links`, `client_logos` 테이블이 생성됐는지 확인
3. 좌측 메뉴 **Authentication** → **Policies** → 각 테이블에 RLS 정책 목록이 있는지 확인
4. 좌측 메뉴 **Database** → **Functions** → `increment_click_count` 함수가 있는지 확인

## 퀴즈 정답

1. RLS를 설정하면 로그인한 사용자만 자신의 데이터를 수정할 수 있도록 제한할 수 있다. → **O**  
   ↳ `auth.uid() = user_id` 조건으로 본인 행만 수정 가능하게 한다.

2. 방문자도 링크를 클릭하면 `click_count`를 직접 UPDATE할 수 있다. → **X**  
   ↳ 방문자는 RLS로 직접 UPDATE할 수 없다. `increment_click_count` RPC 함수를 통해서만 가능하다.

3. 마이그레이션은 DB 구조 변경을 코드로 기록해 이력을 추적하는 방법이다. → **O**  
   ↳ 언제 어떤 테이블과 함수를 만들었는지 이력을 남긴다.
