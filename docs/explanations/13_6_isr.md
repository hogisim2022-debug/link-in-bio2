## 핵심 개념 1: Next.js 렌더링 3가지

Next.js 페이지는 세 가지 방식으로 만들어진다.

```
SSG (Static)    빌드할 때 HTML을 한 번만 만들고 영원히 재사용
ISR             빌드할 때 만들되, N초마다 백그라운드에서 새로 만들 수 있음
SSR (Dynamic)   요청이 올 때마다 매번 새로 만듦
```

**SSG**: 가장 빠름. 내용이 절대 안 바뀌는 페이지에 적합.  
**ISR**: 빠르면서도 주기적으로 최신 데이터 반영 가능. Link-in-Bio에 적합.  
**SSR**: 항상 최신 데이터가 필요한 경우. 느림.

### ISR 목적
빠른 속도는 유지하면서도 데이터는 어느 정도 최신 상태로 유지하려는 것

### 언제 쓰면 좋냐
- 블로그, 뉴스, 상품 페이지처럼 자주 바뀌지만 “매 요청마다 서버 렌더링까지는 필요 없는 경우”
- 트래픽 많은 서비스에서 SSR 부담 줄이고 싶을 때

## 핵심 개념 2: `export const revalidate`

```tsx
// app/page.tsx
export const revalidate = 60; // 60초
```

이 한 줄로 ISR이 활성화된다.

**동작 순서:**
1. 첫 방문자: 빌드 시 만들어진 HTML 즉시 반환 (빠름)
2. 60초 후 누군가 방문: 기존 HTML을 먼저 보여주고, 백그라운드에서 새 HTML 생성 시작
3. 다음 방문자부터: 새로 만들어진 HTML 반환

강사가 Admin에서 링크를 수정해도 최대 60초 안에 공개 페이지에 반영된다.

- ❌ 과거: `"페이지 데이터 자동으로 갱신되게 해줘"`
- ✅ 현재: `"app/page.tsx에 export const revalidate = 60 한 줄 추가해줘"`

## 핵심 개념 3: 지금은 Static인 이유

```
빌드 결과:
├ ○ /        ← Static (ISR 아님)
└ ○ /admin
```

현재 `page.tsx`는 mock 데이터를 import해서 사용한다.  
Next.js가 "이 페이지는 외부 데이터를 fetch하지 않으니 완전 정적"으로 판단한다.  
`revalidate = 60`을 선언해도 fetch가 없으면 ISR로 동작하지 않는다.

Supabase 연동 후 `await fetch(...)` 또는 `createClient().from(...)`으로 데이터를 가져오면 빌드 결과가 `ƒ /` (ISR)로 바뀐다.

## 다음 단계

- Supabase 연동 후 `page.tsx`에서 실제 DB fetch 추가
- `revalidate` 값은 강사가 얼마나 자주 업데이트하는지에 따라 조정
  - 자주 수정: `30`초
  - 거의 안 바뀜: `3600`초 (1시간)
- `next.config.ts`의 Supabase Storage hostname 주석 해제

## 퀴즈 정답

1. ISR은 빌드 시 HTML을 한 번 만들고 이후 N초마다 백그라운드에서 새로 만들 수 있다. → **O**  
   ↳ SSG(완전 정적)와 SSR(매 요청마다 렌더링)의 중간 방식이다.

2. `export const revalidate = 60`을 선언하면 방문자는 항상 최신 데이터를 본다. → **X**  
   ↳ 60초 이내 방문자는 이전에 만든 HTML을 받는다. 60초 후 첫 방문자가 갱신을 트리거한다.

3. mock 데이터를 import하는 지금은 `revalidate = 60`을 선언해도 실제로 ISR로 동작하지 않는다. → **O**  
   ↳ 실제 `fetch` 호출이 없으면 Next.js가 완전 정적으로 판단한다.
