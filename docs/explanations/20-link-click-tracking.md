# 링크 클릭 수 집계 (increment_click_count)

**완료한 task:** `3. Public 페이지 > 3-4. 링크 클릭 시 increment_click_count RPC 호출`  
**수정 파일:** `components/public/LinkItem.tsx`  
**브랜치:** `task/20-link-click-tracking`

---

## 이 문서를 읽고 나면 풀 수 있어요

1. 방문자가 링크를 클릭하면 DB의 `click_count`가 자동으로 1 올라간다.
2. 클릭 수 집계를 위해 링크 이동이 잠시 멈췄다가 이동한다.
3. 방문자(비로그인)는 RLS 때문에 links 테이블을 직접 수정할 수 없다.

---

## 무엇을 했나?

공개 페이지에서 링크를 클릭하면 Supabase에 클릭 수를 기록하도록 연결했다.

```
방문자가 링크 클릭
       ↓
클릭 수 집계 요청 (백그라운드)    →    링크 이동 (동시)
       ↓
Supabase: increment_click_count(링크ID)
       ↓
DB: links.click_count + 1
```

링크 이동과 클릭 수 집계가 동시에 일어나므로 방문자는 지연 없이 이동한다.

---

## 핵심 개념: fire and forget

클릭 수 집계는 **결과를 기다리지 않고 요청만 보내고 바로 이동**한다.

```
일반 방식 (느림)           fire and forget (빠름)
집계 요청 보냄             집계 요청 보냄
     ↓                          ↓ (기다리지 않음)
집계 완료 대기             링크 이동 (즉시)
     ↓
링크 이동
```

집계가 실패해도 방문자 경험에 영향이 없다. 클릭 수는 중요한 데이터지만, 이동을 막을 만큼 중요하지는 않기 때문이다.

- ❌ 과거: `"링크 클릭하면 클릭 수 올려줘"`
- ✅ 현재: `"링크 클릭 시 클릭 수 집계는 백그라운드로 처리하고, 링크 이동은 바로 되게 해줘"`

---

## 이렇게 확인하세요

1. 터미널에서 `npm run dev` 실행
2. 브라우저에서 `http://localhost:3000` 접속
3. 링크 하나 클릭 (새 탭으로 이동됨)
4. Supabase 대시보드 → Table Editor → `links` 테이블 확인
5. 클릭한 링크의 `click_count`가 1 올라갔는지 확인

---

## 퀴즈 정답

1. 방문자가 링크를 클릭하면 DB의 `click_count`가 자동으로 1 올라간다. → **O**  
   ↳ `increment_click_count` RPC 함수가 클릭 시 호출되어 카운트를 올린다.

2. 클릭 수 집계를 위해 링크 이동이 잠시 멈췄다가 이동한다. → **X**  
   ↳ fire and forget 방식이라 집계와 이동이 동시에 일어난다. 방문자는 지연을 느끼지 않는다.

3. 방문자(비로그인)는 RLS 때문에 links 테이블을 직접 수정할 수 없다. → **O**  
   ↳ 그래서 `SECURITY DEFINER` 속성의 RPC 함수를 통해 클릭 수만 올리는 것이다.
