# Task 진행 규칙
- task(`task.md`)는 최소 단위로 하나씩 진행한다.
- task 시작 전: 
    - 해당 task를 대표할 수 있는 간단한 설명과 함께 `git checkout -b task/<번호>-<간단한 설명>`
- task 완료 후 체크리스트 (순서대로):
    1. 설명 문서 작성 (`docs/explanations/<번호>-<설명>.md`)
        - 번호는 기존 문서 중 가장 큰 번호 + 1
        - 최상단: task 메타 정보 + 브랜치명
        - 상단 퀴즈 문제 (정답 없이)
        - 무엇을 했나? + 다이어그램
        - 핵심 개념 + 프롬프트 업그레이드 예시
        - 이렇게 확인하세요
        - 퀴즈 정답
    2. quiz 폴더 복습 파일 작성 (`docs/explanations/quiz/<번호>-<설명>.md`)
    3. 해당 브랜치에 커밋 (커밋 메시지는 한글로)
    4. `git checkout main` → `git merge task/<번호>-<설명>`


# 설명 문서 규칙
## 설명 문서 독자 기준
- 설명 문서(docs/explanations/)의 독자는 **비개발자**다.
- 코드 블록은 개념을 직관적으로 보여줄 때만 쓴다. 로직이 복잡한 코드(조건 분기, async/await, 타입 선언 등)는 코드 대신 흐름도나 비유로 대체한다.
    - ❌ 복잡: `const { data: { user } } = await supabase.auth.getUser();`
    - ✅ 단순: `font-size: 4rem`, `display: flex` 같은 한 줄 속성

## 응답 말투 규칙
- 사용자에게 출력하는 모든 문장에 높임말을 쓰지 않는다.
    - ❌ "완료했습니다.", "진행하겠습니다.", "확인해 주세요."
    - ✅ "완료.", "진행.", "확인 필요."


## 설명 문서 퀴즈 규칙
- **문서 최상단** : task 메타 정보, 생성된 브랜치명

- **문서 상단** (task 메타 정보 바로 아래 `---` 이후): 문제만, 정답 없이
    - 섹션 제목: `## 이 문서를 읽고 나면 풀 수 있어요`
    - OX 형식, 3~5문항, 전체 흐름과 핵심 개념 위주
- **문서 최하단** : 문제 + 정답 + 한 줄 해설을 한 번에 표시
    - 섹션 제목: `## 퀴즈 정답`
    - 형식:
      ```
      1. 문제 텍스트 → **O**
         ↳ 해설
      ```
- **quiz 폴더** (`docs/explanations/quiz/`): 해당 문서와 같은 번호로 복습 파일 생성
    - 앞 문서들과 연계된 심화·함정 문제 위주, 5문항 내외
    - 정답 포함

## 설명 문서 글쓰기 규칙
- "~할 수 있습니다", "~이 가능합니다" 대신 단정형으로 서술한다.
    - ❌ "계산할 수 있습니다."
    - ✅ "계산할 수 있다."
- 설명 문서(docs/explanations/)를 작성할 때 "무엇을 했나?" 섹션 아래에 다이어그램을 추가한다.
    - **UI task**: ASCII 화면 구조도 — 실제 화면에서 어떻게 보이는지를 박스/선으로 표현한다. 흐름도가 아닌 화면 레이아웃 중심으로 그린다.
    - **백엔드 task**: ASCII 흐름도 또는 데이터 흐름도 — 요청/응답 흐름, DB 처리 순서, 인증 분기 등을 화살표와 박스로 표현한다.
    - **이외의 task**: 관련 diagram을 그리고 설명한다.
- 설명 문서에 `## 이렇게 확인하세요` 섹션을 추가한다.
    - 위치: `## 퀴즈 정답` 바로 위
    - task 타입에 따라 내용을 다르게 작성한다. 예시 참고.
        - **UI task** (컴포넌트, 화면 구성): `npm run dev` → 브라우저 URL + 클릭/입력 순서 + 눈으로 확인할 것
        - **DB task** (테이블, RLS, RPC): Supabase 대시보드 → Table Editor 또는 SQL Editor에서 데이터·정책 확인
        - **Storage task** (버킷, 파일 업로드): Supabase 대시보드 → Storage에서 버킷·파일 확인
        - **인증 task** (로그인, 로그아웃, Middleware): `npm run dev` → 브라우저에서 리다이렉트 동작 + 쿠키 확인
        - **데이터 연동 task** (mock → Supabase 교체): `npm run dev` → 페이지에서 실제 DB 데이터가 표시되는지 확인
        - **설정 task** (환경변수, 패키지, next.config): `npm run dev` 오류 없이 실행되는지 + 관련 설정 파일 내용 확인
        - **성능/배포 task** (ISR, Vercel, Lighthouse): `npm run build` 결과 확인 또는 배포 URL에서 동작 확인
        - **위 유형에 해당하지 않으면**: task 결과물을 직접 눈으로 확인할 수 있는 가장 짧은 경로를 서술한다.
    - 비개발자도 따라할 수 있도록 단계별로 서술한다.
    - 개발 서버 실행이 필요한 경우 첫 번째 단계로 포함한다.
- 핵심 개념 섹션 중 프롬프트 작성에 직접 활용 가능한 개념이 있으면, **그 섹션 마지막**에 프롬프트 전후 예시를 추가한다.
    - 문서 끝이 아니라 해당 개념 설명 바로 아래에 붙인다.
    - 형식 (볼드 없이, 바로 불릿):
      - ❌ 과거: `"여기 폰트 좀 키워줘"`
      - ✅ 현재: `"h1 font-size를 4rem으로 바꿔"`
    - 개념과 무관한 일반 작업 지시 섹션에는 넣지 않는다.
